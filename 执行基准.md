## 3. X 账号解析与抓取

### 3.1 X 身份解析优先级

按以下顺序解析目标账号：

1. `feeRecipient.xUsername`
2. `deployer.xUsername`
3. `tweetUrl` 是 X 主页时，从 URL 提取 username
4. `tweetUrl` 是 X status 时，用 `fxtwitter` status detail 反查 `author.screen_name`
5. 全部失败则标记为 `unresolved`

需要给身份来源打标签：

- `fee_recipient_x_username`
- `deployer_x_username`
- `tweet_url_profile`
- `tweet_url_status_author`
- `unresolved`

### 3.2 X 历史抓取目标

已定目标：

- 每个账号尽力抓取 `50` 条历史发言。
- 抓不到 50 条时返回实际数量，并记录原因。

默认抓取策略：

- `target_tweets = 50`
- `max_pages = 10`
- `count = 100`
- 通过 `cursor.bottom` 连续翻页。
- 按 `tweet_id` 去重。
- 仅保留 `author.screen_name == target_username` 的记录。

实测结论：

- 活跃个人号通常能抓满 50 条。
- 新号、项目号、低活跃号经常不足 50 条。
- 不能假设单页会返回 100 条，`fxtwitter` 实测常见单页约 20 条。

## 4. 账号评估方向

核心不是简单判断“强不强”，而是判断这个 fee recipient 是否可能接受 permissionless token，以及他是否有开发/传播价值。

### 4.1 评估维度

前端和 AI 分析使用以下维度：

- `Crypto Affinity`
- `Token Receptivity`
- `Builder Signal`
- `Open-source Footprint`
- `Social Leverage`
- `Risk`
- `Confidence`

### 4.2 账号类型

模型需要输出账号类型，例如：

- `crypto_native_builder`
- `ai_builder`
- `trader_or_kol`
- `project_account`
- `general_tech_builder`
- `non_crypto_public_figure`
- `unknown`

### 4.3 最终判断

模型需要输出 token 接受倾向：

- `likely_accept`
- `likely_tolerate`
- `unclear`
- `likely_reject`

### 4.4 分析原则

- `开发能力` 和 `Token Receptivity` 是两条独立轴。
- 强开发者不一定接受被发币。
- Crypto 项目号也不一定有强传播力。
- `launch.websiteUrl` 里的 GitHub repo 不一定属于 X 账号本人，需要区分来源。

GitHub 来源分类：

- `owned_repo`
- `profile_repo`
- `mentioned_repo`
- `launch_website_repo`

## 6. 前端产品方向

### 6.1 总体风格

已定：

- Apple 风格基底。
- 简洁、清晰、留白足。
- 内容表达可以更酷、更有科技感。
- 不做营销 landing page，首页就是可用的数据看板。

视觉关键词：

- clean glass
- precise typography
- subtle motion
- high-contrast data visualization
- calm but futuristic

### 6.2 主要页面

第一版建议只有一个核心看板页面：

- 左侧或顶部：最新 token feed
- 中央：当前选中 token / account 的评估详情
- 右侧或下方：分析过程、证据、推文/GitHub 来源

### 6.3 每个账号展示内容

账号卡片/详情需要展示：

- token 基础信息：ticker、CA、chain、launch time
- fee recipient：wallet、X username、身份来源
- X profile：bio、followers、following、statuses、website
- 最近推文抓取量：例如 `42 / 50`
- 互动统计：median likes、median reposts、median replies、median views
- GitHub 信息：repo/user 来源、stars、forks、contributors、pushed_at、license
- AI 总结：英文展示
- AI 分析理由：英文展示，点击展开
- 风险和置信度

### 6.4 数据可视化

需要突出“水平”的视觉表达。

第一版图表建议：

- 横向柱状图：展示 6 个账号评分维度。
- 小型趋势/条形组合：展示最近推文互动中位数。
- 状态进度条：展示抓取进度，例如 `Fetching X history 32/50`。
- 置信度 ring 或 slim meter：展示 `Confidence`。

### 6.5 动效与实时感

前端需要向用户展示系统正在工作：

- `Searching for new tokens`
- `Resolving recipient identity`
- `Fetching X history`
- `Scanning GitHub signals`
- `Thinking about account fit`
- `Writing final assessment`

建议实现方式：

- 后端把每个 launch/account 的 processing stage 写入 Supabase。
- 前端订阅或轮询 Supabase 最新状态。
- 先实现轮询，后续可升级 Supabase Realtime。

第一版前端刷新策略：

- 每 5 秒刷新 dashboard 数据。
- 每个 account 显示当前 stage 和 partial result。
- AI 分析完成前展示 skeleton、进度文字和流动高亮。

关于“前端流式生成”：

- 真正 token-level streaming 到 Vercel 前端会增加复杂度。
- 当前最终数据存 Supabase，第一版建议使用 `stage + partial_result` 模拟流式体验。
- 后端可分阶段写入 profile、tweets、github、analysis。
- 前端每次刷新看到字段逐步补齐，用户感知上接近实时生成。
