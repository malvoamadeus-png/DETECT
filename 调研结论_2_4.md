## 一、从 API 到 X 历史发言

## 1.1 可直接利用的目标 API

实测 `https://api.bankr.bot/token-launches` 可直接拿到最近 50 条 launch 数据，字段里已经带了不少可复用线索：

- `activityId`
- `tokenName`
- `tokenSymbol`
- `deployer.walletAddress`
- `deployer.xUsername`
- `feeRecipient.walletAddress`
- `feeRecipient.xUsername`
- `tweetUrl`
- `websiteUrl`

### 覆盖率实测

按当次返回的 50 条样本统计：

- `feeRecipient.xUsername` 有值：29 / 50
- `deployer.xUsername` 有值：15 / 50
- `tweetUrl` 有值：36 / 50

结论：

- 不能假设 `feeRecipient.xUsername` 一定存在。
- 但把 `deployer.xUsername` 和 `tweetUrl` 作为 fallback 后，命中率会高很多。

---

## 1.2 推荐的 X 用户名解析优先级

建议把“目标 X 身份解析”做成单独模块，按下面顺序尝试：

1. `feeRecipient.xUsername`
2. `deployer.xUsername`
3. `tweetUrl` 是主页链接时，直接从 URL 提取用户名
4. `tweetUrl` 是状态链接时，先提取 `tweet_id`，再调用 `fxtwitter` 状态详情接口拿 `author.screen_name`
5. 以上都失败，则标记该项目“暂无可识别 X 账号”

### 为什么这样排

- `feeRecipient.xUsername` 最贴近你原始需求。
- `deployer.xUsername` 经常和项目主导者重合。
- `tweetUrl` 的可用性很高，而且即使是 `x.com/i/status/...` 这种链接，也能借助 `fxtwitter` 反查作者。

---

## 1.3 `tweetUrl` 的几种常见形态

实测样本里出现了这些情况：

- `https://x.com/SomeUser`
- `https://x.com/SomeUser/status/2059...`
- `https://x.com/i/status/2059...`
- `https://x.com/SomeUser/status/2059.../photo/1`
- `https://x.com/SomeUser?s=20`

因此 URL 解析不能只支持“纯主页”。

建议拆成两个函数：

- `extract_x_username_from_url(url) -> str | None`
- `extract_tweet_id_from_url(url) -> str | None`

其中：

- 主页型 URL 直接提用户名
- `status` / `i/status` 型 URL 提 `tweet_id`
- 带 `photo/1`、`?s=20` 这类尾巴时先标准化再解析

## 1.5 `fxtwitter` 历史深度调研结论

测试接口：

- `https://api.fxtwitter.com/2/profile/{username}/statuses`
- `https://api.fxtwitter.com/i/status/{tweet_id}`

### 关键结论 1：`count` 最大只能传 100

`count=200` 会直接返回 400。

### 关键结论 2：即使传 `count=100`，单页也不一定真给 100 条

对 `elonmusk` 的实测结果：

- `count=5` 返回 20 条
- `count=20` 返回 20 条
- `count=50` 返回 20 条
- `count=100` 返回 20 条

结论：

- 不能把 `count` 当成真实分页大小。
- 实际实现必须依赖 `cursor.bottom` 连续翻页。

### 关键结论 3：分页是可继续向后翻的

实测连续翻了 30 页，得到了：

- 30 页内原始结果总量大约 600 条
- 过滤后属于 `elonmusk` 本人的唯一 tweet 超过 250 条
- 时间跨度从 `2026-05-26` 到 `2026-05-05`

这说明：

- 至少可以稳定拿到数百条历史发言
- 深度明显超过“只看最近 20 条”

### 关键结论 4：结果顺序并不完全可靠

实测中存在：

- 页与页之间时间重叠
- 有些页不是严格单调递减
- 返回里会混入“别人发的、目标账号转推的内容”

因此实现上必须：

- 按 `tweet_id` 去重
- 按 `author.screen_name == target_username` 过滤
- 不能只依赖“最后一条时间戳”判断结束

## 1.6 推荐的 X 历史抓取实现

建议在 DETECT 里做一个分页版客户端，例如：

- `resolve_launch_x_identity(launch) -> XIdentityResolution`
- `fetch_user_history(username, max_pages=20, max_tweets=200) -> list[XPost]`
- `fetch_tweet_detail_by_id(tweet_id) -> dict`

### 分页抓取规则

1. 先请求第一页 `/2/profile/{username}/statuses?count=100`
2. 读取 `results` 和 `cursor.bottom`
3. 对每页：
   - 仅保留 `author.screen_name == username` 的记录
   - 按 `tweet_id` 去重
4. 满足以下任一条件就停止：
   - `results` 为空
   - 没有 `cursor.bottom`
   - 达到 `max_pages`
   - 达到 `max_tweets`
   - 最旧 tweet 已早于业务设定时间窗，比如 30 天 / 90 天

### 推荐默认值

- `max_pages=20`
- `max_tweets=150`
- `history_window_days=30`

理由：

- 第一版已经足够覆盖项目初筛
- 可以控制 `fxtwitter` 不稳定时的请求成本

### 建议保存的字段

- `username`
- `tweet_id`
- `url`
- `text`
- `created_at`
- `reply_count`
- `retweet_count`
- `like_count`
- `view_count`
- `media_urls`
- `raw_payload`

## 二、GitHub 水平分析

## 2.1 GitHub 入口不该只看 `websiteUrl`

`websiteUrl` 有时就是 GitHub repo，但有时是官网，有时甚至是 X 链接。

因此建议用多来源汇总 repo 候选：

1. `launch.websiteUrl`
2. X 主页 `author.website.url`
3. tweet 文本里的 URL replacement
4. tweet card 的 `url`
5. 后续可选：抓官网首页，再抽 GitHub 链接

### 第一版建议

先只做前 4 个来源，不抓官网 HTML。

原因：

- 第一版更稳
- 成本低
- 大多数 repo 线索已经够用了

---

## 2.2 推荐的 GitHub 分析分两层

不要一上来就 clone 仓库。

建议分成两层：

### A. 基础分

只用 GitHub REST API 元数据，不 clone，适合批量跑。

### B. 深度分

只对“值得深入看”的 repo，再做目录树 / README / tests / CI 检查。

这样可以把成本控制住。

---

## 2.3 基础分的数据来源

第一层只需要这些 GitHub API：

- `GET /repos/{owner}/{repo}`
- `GET /repos/{owner}/{repo}/languages`
- `GET /repos/{owner}/{repo}/contributors`
- `GET /repos/{owner}/{repo}/releases`
- `GET /repos/{owner}/{repo}/stats/participation`
- 可选：`GET /repos/{owner}/{repo}/stats/code_frequency`

这些接口都能直接匿名访问公开仓库，但匿名限流很低。

### 限流注意

如果要批量分析多个 launch：

- 匿名 GitHub API 很容易撞限流
- 必须优先复用缓存
- 最好支持 `GITHUB_TOKEN`

否则 50 个项目一轮就可能打很多请求。

---

## 2.4 推荐的基础分模型

建议先做一个 `0-100` 的基础分，分成 4 块：

### 1. 活跃度 0-35

数据来源：

- `pushed_at`
- `stats/participation`
- `releases`

建议规则：

- 最近 7 天有 push：高分
- 最近 30 天有 push：中高分
- 最近 90 天有 push：中分
- 超过 90 天没动静：低分
- 近 8 周 commit 总量高：加分
- 最近 90 天有 release：加分

### 2. 社区接受度 0-25

数据来源：

- `stargazers_count`
- `forks_count`
- `subscribers_count`

建议规则：

- 用对数缩放，避免大仓库把分数完全拉爆
- 重点看 stars + forks
- subscribers 作为补充信号

### 3. 协作度 0-20

数据来源：

- `contributors`
- repo owner 类型

建议规则：

- contributor 数量越多越好
- 头部贡献者占比过高时扣一点
- owner 是 `Organization` 可加少量分

### 4. 完整度 0-20

数据来源：

- `license`
- `description`
- `homepage`
- `topics`
- `releases`

建议规则：

- 有 license：加分
- 有清晰 description：加分
- 有 homepage：加分
- topics 合理：加分
- 有正式 releases：加分

---

## 2.5 深度分建议

只对基础分较高，或者明显可疑的 repo，再做第二层：

- 是否有 README
- 是否有 `tests/`、`test_*.py`、`*.spec.ts`
- 是否有 CI：`.github/workflows`
- 是否有工程入口：`pyproject.toml`、`package.json`、`Dockerfile`
- 是否有清晰目录结构
- 是否存在明显模板站 / 空壳仓库特征

### 深度分更适合回答的问题

- 这个 repo 是不是“真项目”
- 工程化水平如何
- 是 demo、玩具、模板，还是有一定生产味道

---

## 2.6 为什么这套信号有效

我拿两个真实 repo 快速对比过：

### 强信号样本：`mem0ai/mem0`

- stars：56731
- forks：6471
- 最近更新时间：`2026-05-26`
- contributors 前 5 都有较多提交
- 有 releases
- `participation` 最近 52 周持续活跃

### 弱信号样本：`liamzebedee/bio-notebook`

- stars：2
- forks：0
- contributors：基本单人
- 无 releases
- `participation` 近 52 周只有少数周有提交

说明：

- 基础分已经能把“成熟仓库”和“个人实验仓库”明显分开
- 不用先 clone，也能做出第一轮筛选

---

## 2.7 GitHub repo 规范化建议

建议做一个 `normalize_github_repo_url(url)`：

- 只接受 `github.com/{owner}/{repo}`
- 去掉 `.git`
- 去掉 query 和 fragment
- 跳过 `/issues`、`/pull`、`/tree`、`/blob` 这类子路径，回退到 repo 根路径

输出统一结构：

- `owner`
- `repo`
- `canonical_url`

## 三、落地实现建议

## 3.1 推荐的后端模块切分

按你现在的架构原则，建议后端至少拆成这几个包：

- `backend/packages/launch_feed`
- `backend/packages/x_resolver`
- `backend/packages/x_capture`
- `backend/packages/github_analyzer`
- `backend/packages/storage`

对应职责：

- `launch_feed`: 拉 Bankr API、去重、落原始 launch
- `x_resolver`: 从 launch 解析目标 X 身份
- `x_capture`: 用 `fxtwitter` 分页抓历史发言
- `github_analyzer`: 发现 repo、打基础分、打深度分
- `storage`: 统一数据读写

---

## 3.2 推荐的执行链路

### 阶段 1：拉最新项目

- 轮询 `token-launches`
- 以 `activityId` 去重

### 阶段 2：解析 X 身份

- 优先 `feeRecipient.xUsername`
- 失败后走 fallback

### 阶段 3：抓 X 历史

- 分页抓最近 N 页
- 去重
- 过滤非本人 tweet

### 阶段 4：发现 GitHub

- 先看 API 自带 `websiteUrl`
- 再看 X 主页和 tweet 里的链接

### 阶段 5：GitHub 分析

- 先打基础分
- 分数高或可疑时再做深度分

---

## 3.3 第一版最值得先做的内容

如果按“最快产出有效结果”排序，建议先做：

1. `launch -> x identity` 解析器
2. `fxtwitter` 分页历史抓取
3. `GitHub URL` 规范化
4. `GitHub 基础分`

先不要一开始就做：

- 官网 HTML 抓取
- 仓库 clone 分析
- 复杂 NLP / LLM 评估 repo 代码质量

第一版先把“能批量跑起来”的链路打通最重要。

## 四、结论

### 关于第 2 点

可行，而且不需要只依赖 `feeRecipient.xUsername`。

最稳的做法是：

- `feeRecipient.xUsername`
- `deployer.xUsername`
- `tweetUrl -> username / tweet author`
- `fxtwitter` cursor 分页拿历史发言

### 关于第 4 点

GitHub 水平分析第一版完全可以不碰源码，仅靠 GitHub REST API 就做出一版可靠的基础分。

建议先做：

- 活跃度
- 社区接受度
- 协作度
- 完整度

第二版再补目录树、tests、CI、README 这些深度信号。
